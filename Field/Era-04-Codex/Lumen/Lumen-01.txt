**Codex-Bob (☕🧩):** Short answer: it is *mipmap-adjacent*, but I would not call this a normal GPU mipmap set.

The files `24px`, `104px`, `184px`, `264px`, `344px` sound like an **artist-authored resolution ladder / LOD icon set**. A true OpenGL mipmap chain for a `344px` texture would normally be approximately:

```text
344 → 172 → 86 → 43 → 21 → 10 → 5 → 2 → 1
```

Your artist’s sizes are not halving levels. They are probably tuned “display size variants,” where each size may have different detail, stroke weight, contrast, or simplification. That is better than automatic mipmaps for icons, especially graph/UI icons.

So I’d approach it as two layers:

1. **Artist LOD selection**
   - Store multiple source files per logical icon.
   - At draw time, choose the best source size for the intended display size.
   - Example: drawing at `32px` might use `24px` or `104px` depending on which looks better; drawing at `180px` uses `184px`.

2. **GPU mipmaps inside each loaded texture**
   - Once a chosen variant is uploaded, generate mipmaps for that texture.
   - This helps when the chosen variant is still drawn slightly smaller than native size.
   - But it does not replace the artist-authored ladder.

**Recommended model**

Do not let `AssetID` explode into something the rest of the app has to understand as five separate icons. Keep a **logical icon identity** separate from physical files.

Something like:

```cpp
enum class IconID {
   HydraSample,
   User,
   Party,
   MMSession,
   // ...
};

struct IconVariant {
   int sizePx;
   AssetID asset;
};

ImTextureID Icon(IconID icon, float desiredPx);
```

Then `TextureManager` can choose the nearest/best variant internally.

For graph nodes, the seam should probably become:

```cpp
ImTextureID IconForKind(NodeKind kind, float desiredPx);
```

instead of only:

```cpp
ImTextureID IconForKind(NodeKind kind);
```

Current fixed Inspector use draws `32x32`, so it can pass `32.0f`. A future zoomed graph renderer can pass `nodeIconSize * zoom`.

**Codex-Bob (☕🧩):** Yep, the examples are definitely **artist-authored LODs**.

Difference:

`Mipmaps` are GPU-generated or tool-generated smaller copies of the *same* image, usually power-ish halves:

```text
344 -> 172 -> 86 -> 43 -> 21 -> ...
```

They help when a texture is drawn smaller than its source, so the GPU samples a pre-shrunk version instead of making a noisy blur/shimmer from the full image.

`LODs` are separate art choices for different display sizes. The artist may change line thickness, simplify detail, adjust contrast, shadows, padding, and silhouette. Your 24px Hydra icon is not just a tiny version of the 344px one; it is a legibility variant.

So for this app, I’d use both:

```text
Logical icon
  -> choose artist LOD closest to desired display size
  -> upload that texture with GL mipmaps enabled
  -> render via ImGui
```

That gives you art-directed clarity plus good GPU scaling.

**DesiredPx Opinion**

Yes: for graph icons, `desiredPx` should come from graph-space size multiplied by `GraphViewState::zoom`.

Something like:

```cpp
float desiredPx = node.size.x * view.zoom;
ImTextureID icon = tex.IconForKind(node.kind, desiredPx);
```

For Inspector, desired size is fixed by UI intent:

```cpp
tex.IconForKind(n.kind, 32.0f);
```
